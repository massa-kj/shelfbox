# v0.9.0 Compatibility Baseline

This document records the symlink-mode behavior and public Rust surface that
copy mode must preserve. It is a Phase 0 baseline, not a copy-mode design
document. The normative copy-mode contract is
[`../spec/copy-mode.md`](../spec/copy-mode.md).

The former draft is historical input only. Code, tests, and documentation must
cite the normative specification under `docs/spec/`.

## Pre-change test baseline

On 2026-07-04, before Phase 0 edits, `cargo test --workspace` passed on the
Linux development host:

| Suite | Result |
|---|---:|
| `shelfbox` CLI behavior locks | 16 passed |
| `shelfbox-core` unit and integration tests | 244 passed |
| Binary unit tests and doc tests | 0 failed |

The repository CI workflow runs `cargo check`, strict Clippy, and the complete
workspace suite on Ubuntu, macOS, and Windows. Cross-platform baseline evidence
for a change is complete only when all three matrix jobs pass; a local Linux
run does not substitute for that evidence.

## Executable behavior locks

The following pre-copy behavior is intentionally locked:

| Contract | Evidence |
|---|---|
| Healthy, warning-only, and error item-status exit codes are `0`, `1`, and `2` | `item_status_exit_codes_are_locked_for_associated_repo` |
| Healthy, warning-only, and error repo-status exit codes are `0`, `1`, and `2` | `repo_status_exit_codes_are_locked_for_associated_repo` |
| Healthy symlink item JSON retains its fields and boolean values | `fixtures/item-status-symlink-v1.json` |
| Healthy symlink repo JSON retains its outer object and item values | `fixtures/repo-status-symlink-v1.json` |
| Wrong-target repair refuses without force and relinks with force | `repair_rejects_wrong_target_symlink_without_force`, `repair_force_relinks_wrong_target_symlink` |
| `restore --keep-store` retains the manifest item, store item, and symlink | `restore_keep_store_leaves_symlink_and_store_item` |
| GC collects only orphaned ownership and preserves unreachable data | GC policy, operation, CLI, and recovery tests |
| Rebuild-index reconstructs cache data without inventing Git association | rebuild-index operation, CLI, and recovery tests |
| Reclaim requires explicit selection and does not repair symlinks | reclaim operation, CLI, scenario, and recovery tests |

## JSON v1 baseline

The legacy item status object has these required fields and meanings:

```text
path: String
link_exists: bool
link_valid: bool
store_exists: bool
in_exclude: bool
not_tracked: bool
ok: bool
```

`item status --format json` serializes an array of those objects. `repo status
--format json` serializes an object with `items`, `orphan_store_items`, and
`repo_root_matches_index`. Phase 4 must preserve the existing symlink values
while introducing copy-aware v2 DTOs.

## Public Rust compatibility inventory

The external-crate fixture at
`crates/shelfbox-core/tests/public_api_compat.rs` compiles against exact
directionless function signatures and constructs representative status,
report, plan, and error values with struct literals. The following existing
symlink-specific names require compatibility treatment when internals become
materialization-generic:

| Surface | Legacy names that must remain usable in v0.9.0 |
|---|---|
| `api::item::ItemStatus` | `link_exists`, `link_valid` boolean fields |
| `api::item::ItemInfo` | `link_target`, `symlink_ok` fields |
| `plan::item_relink::ItemRelinkPlan` | `symlink_ok` field |
| `plan::repo_repair::RepoRepairSymlinkAction` | exported enum and all existing variants |
| `plan::repo_repair::RepoRepairPlan` | `symlink_actions` field |
| `plan::repo_repair::RepairRepoReport` | `symlinks_repaired`, `symlinks_already_healthy`, `symlinks_failed` fields |
| `error::AppError` | `PathIsSymlink`, `MoveSourceSymlinkMismatch`, `RepairSymlinkTargetMismatch`, and `NotManagedLink` variants |

The directionless APIs protected by exact function types include item add,
restore, repair, relink, move, status, info, repo status/repair/reclaim, store
GC/rebuild/migration, and configuration access. Copy-aware options must be
introduced through additive request types, variants, or v2 entry points rather
than silently changing these signatures.

The plan/report field sets protected by construction include:

* `ItemAddPlan` / `ItemAddReport`
* `ItemRestorePlan` / `ItemRestoreReport`
* `ItemRelinkPlan` / `ItemRelinkReport`
* `ItemMovePlan` / `ItemMoveReport`
* `ItemRepairReport`
* `RepoRepairPlan` / `RepairRepoReport`
* `GcCandidate` / `GcPlan` / `GcReport`
* `RebuildIndexWarning` / `RebuildIndexReport`

Adding a field to a publicly constructible Rust struct is source-breaking for
downstream struct literals. Generic materialization data therefore belongs in
dedicated v2 DTOs or other explicitly additive compatibility boundaries.

## Approved v0.9.0 compatibility changes

The implementation plan identifies these intentional changes:

* Dedicated v2 status/report DTOs gain generic materialization fields.
* V2 JSON link fields are nullable for copy items while symlink values remain
  unchanged.
* Copy plus a missing exclude is an error.
* `restore --keep-store` always preserves the exclude entry and is documented
  as legacy detach behavior.
* Repo repair protects detached items that retain repo entries, and preserves
  but does not newly add excludes for detached-plus-missing items.

No manifest schema change is approved for copy mode.
